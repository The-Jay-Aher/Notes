# Professional Trading Workbook Methodology

## Quick Summary

A professional trading workbook is not a diary. It is a research, execution, risk, and review system. Its purpose is to make every strategy explicit, every trade attributable, every mistake visible, and every promotion from idea to live trading evidence-based.

The workbook should answer four questions:

1. What do I believe?
2. What evidence supports it?
3. How exactly do I trade it?
4. What does the data say after execution?

---

# 1. Strategy Playbook

Purpose: Store only tradeable strategies with complete rules.

Fields:

| Field | Purpose |
| --- | --- |
| Strategy name | Unique identifier |
| Hypothesis | Why edge should exist |
| Market/instrument | Where it applies |
| Regime | When it should work |
| Entry/exit/stop | Exact operating rules |
| Sizing | Risk model |
| Evidence | Backtest/forward-test links |
| Status | Idea, test, live-small, retired |

How it is used: No trade is allowed unless it maps to a strategy playbook entry.

---

# 2. Setup Library

Purpose: Maintain examples of valid, invalid, winning, losing, and skipped setups.

Fields:

| Field | Purpose |
| --- | --- |
| Setup name | Pattern identifier |
| Screenshot before | Decision-time evidence |
| Screenshot after | Outcome evidence |
| Valid/invalid | Classification |
| Notes | What made it clean or weak |

How it is used: Builds pattern recognition without hindsight rewriting.

---

# 3. Market Regime Map

Purpose: Stop applying the same strategy in every market.

Fields:

| Field | Purpose |
| --- | --- |
| Date | Review period |
| Index trend | Bull, bear, range, transition |
| Breadth | Broad or narrow participation |
| Volatility | Low, normal, high |
| Sector leadership | Where money is flowing |
| Allowed strategies | What can be traded |

How it is used: Strategy selection starts with regime, not with a random chart.

---

# 4. Watchlist Process

Purpose: Convert market scanning into an actionable list.

Fields:

| Field | Purpose |
| --- | --- |
| Symbol | Tradable candidate |
| Sector | Exposure grouping |
| Setup | Why it is watched |
| Trigger | Exact entry condition |
| Stop | Planned invalidation |
| Liquidity | Whether trade is executable |
| Event risk | Earnings/news risk |

How it is used: Prepares trades before market pressure begins.

---

# 5. Pre-Market Plan

Purpose: Define the day before emotion enters.

Fields:

| Field | Purpose |
| --- | --- |
| Market regime | Bias |
| Key levels | Where price can react |
| News/events | Risk controls |
| Watchlist triggers | Planned entries |
| Daily risk limit | Stop point |
| Mistake to avoid | Behavioral guardrail |

How it is used: If the trade was not in the plan, it needs a high burden of proof.

---

# 6. Daily Trading Plan

Purpose: Translate the pre-market view into allowed decisions.

Fields:

| Field | Purpose |
| --- | --- |
| Allowed setups | What can be traded |
| No-trade zones | Where not to act |
| Scenario plan | If/then conditions |
| Max trades | Overtrading control |
| Max loss | Damage limit |

How it is used: The trader follows scenarios, not predictions.

---

# 7. Risk Sheet

Purpose: Make risk visible before trades are placed.

Fields:

| Field | Purpose |
| --- | --- |
| Capital | Base for sizing |
| Risk per trade | Planned loss |
| Open risk | Total portfolio heat |
| Sector exposure | Concentration |
| Correlation | Hidden duplication |
| Drawdown | Current pain |
| Kill switch | Stop condition |

How it is used: No new trade if portfolio risk breaches limits.

---

# 8. Execution Checklist

Purpose: Prevent avoidable order-entry mistakes.

Fields:

| Field | Purpose |
| --- | --- |
| Order type | Market/limit/stop |
| Quantity | Sizing confirmation |
| Spread | Execution cost |
| Liquidity | Fill realism |
| Stop order | Protection |
| Event check | Gap risk |

How it is used: Execution is treated as a process, not a button press.

---

# 9. Trade Journal

Purpose: Store complete trade evidence.

Fields:

| Field | Purpose |
| --- | --- |
| Trade ID | Traceability |
| Setup | Strategy attribution |
| Entry/exit | Execution record |
| Planned R | Risk basis |
| Actual R | Performance |
| MFE/MAE | Stop/target evaluation |
| Mistake tag | Process improvement |

How it is used: Journal data feeds weekly/monthly reviews.

---

# 10. Post-Trade Review

Purpose: Separate outcome from process.

Categories:

| Category | Meaning |
| --- | --- |
| Good win | Followed rules, won |
| Good loss | Followed rules, lost |
| Bad win | Broke rules, won |
| Bad loss | Broke rules, lost |

How it is used: Bad wins are treated as process failures, not success.

---

# 11. Weekly Review

Purpose: Detect short-term process drift.

Fields:

| Field | Purpose |
| --- | --- |
| Expectancy | Edge estimate |
| Rule-following rate | Discipline |
| Setup performance | What works |
| Mistake log | What repeats |
| Next-week rule | One correction |

How it is used: Weekly review changes behavior, not the entire system after one loss.

---

# 12. Monthly Performance Review

Purpose: Decide whether strategies deserve more, less, or no capital.

Fields:

| Field | Purpose |
| --- | --- |
| Return | Result |
| Drawdown | Risk |
| Profit factor | Gross win/loss quality |
| Regime performance | Context |
| Costs | Friction |
| Strategy status | Promote/demote/retire |

How it is used: Live size changes only after process and data support it.

---

# 13. Mistake Log

Purpose: Convert repeated errors into explicit rules.

Fields:

| Field | Purpose |
| --- | --- |
| Mistake | What happened |
| Trigger | Why it happened |
| Cost | Damage |
| Prevention rule | Fix |
| Recurrence count | Severity |

How it is used: The same mistake repeated three times becomes a system design problem.

---

# 14. Hypothesis Backlog

Purpose: Store ideas without trading them prematurely.

Fields:

| Field | Purpose |
| --- | --- |
| Hypothesis | Research idea |
| Reason | Why it might work |
| Data required | What is needed |
| Test design | How to validate |
| Priority | Research order |

How it is used: Ideas enter research before they enter the account.

---

# 15. Backtest Tracker

Purpose: Track research quality.

Fields:

| Field | Purpose |
| --- | --- |
| Strategy | What was tested |
| Data range | Sample |
| Costs | Realism |
| Bias checks | Validity |
| Metrics | Results |
| Decision | Reject/test/live-small |

How it is used: Backtests are reviewed as evidence, not accepted as proof.

---

# 16. Forward-Test Tracker

Purpose: Check whether backtest behavior survives live market conditions.

Fields:

| Field | Purpose |
| --- | --- |
| Trade count | Sample |
| Rule adherence | Execution quality |
| Slippage | Realism |
| Drawdown | Risk |
| Expectancy | Early evidence |
| Promotion decision | Next step |

How it is used: No strategy moves to live size without forward-test evidence.

---

# 17. Metrics Dashboard

Purpose: Show performance at a glance.

Core metrics:

* Expectancy.
* Profit factor.
* Win rate.
* Average win/loss.
* Max drawdown.
* Rule-following rate.
* Cost per trade.
* Setup performance.
* Regime performance.

---

# 18. Risk Dashboard

Purpose: Prevent hidden concentration and drawdown escalation.

Core metrics:

* Open risk.
* Gross exposure.
* Net exposure.
* Sector exposure.
* Correlated exposure.
* Largest position.
* Current drawdown.
* Distance to kill switch.

---

# 19. Lessons Learned Database

Purpose: Turn experience into reusable knowledge.

Fields:

| Field | Purpose |
| --- | --- |
| Lesson | What changed |
| Source | Trade, review, backtest |
| Rule impact | What rule changed |
| Date | Traceability |
| Status | Active/retired |

How it is used: Learning is captured as rules, not vague memory.
